最近刚开始摸索响应式的代码的设计方案以及移动端的app的界面设计，于是自然冒出了对单位进行一个新的且全面的认识的想法。

css中的单位主要分为两类（此分类来自《css权威指南》第三版）：

绝对长度单位：in,cm,mm,pt,pc(应用场景很少，所以…不作为本篇重点)

相对单位长度：px,em,ex,remcss3属性，IE678（你懂的）

在浏览其他关于css单位的文章，对于px的分类，还有不同的说法，对于像素这个非自然长度单位，它的大小取决于浏览器的分辨率的问题，所以在command+/command-(windows 平台下则是ctrl+/ctrl-)的操作下，可能会出现布局的混乱的问题。原有的布局被打乱，这个时候，做响应式的布局或是对单位选择和应用，就可以有效避免这样的问题。

关于px：

px基本算是最常用的单位，对于刚开始接触css的单位的人来说，就算不去深入了解css的单位的内容，也会自然的上手使用。简单，至关以及精确，基本上可以解决问题。

但是，要考虑到兼容与实际的使用时，px作为一个相对单位，会随着屏幕的分辨率的改变，而出现不同的大小效果，所以会造成隐形伤害，及上部举出的反面案例，commamd（ctrl）+/-的问题；

这个时候，就有人提出用em来救场子了；

关于em：

关于像素，某种程度上px就是一个绝对单位；而em则是基于浏览器的字体大小的相对单位，它相对父节点的font-size而决定大小。一般来说，浏览器的默认字体大小为16px；即1em = 16px;以此类推18px = 1.125em…当然，要时刻揣着计算器来计算相对大小使用em，相当麻烦；
